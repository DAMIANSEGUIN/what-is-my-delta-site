### üî¥ CRITICAL (Production Down)

| Error Label | Symptom | Root Cause | First Action |
|-------------|---------|------------|--------------|
| `RAILWAY_RESTART_LOOP` | Container crashes repeatedly | Code bug, dependency missing | Check deploy logs for exception |
| `PG_CONNECTION_FAILED` | App using SQLite fallback | Wrong DATABASE_URL, network issue | Verify DATABASE_URL contains `railway.internal` |
| `CONTEXT_MANAGER_BUG` | AttributeError: 'object has no attribute execute' | Using `conn = get_conn()` instead of `with get_conn() as conn:` | Search codebase for incorrect pattern |
| `OPENAI_INVALID_KEY` | All AI features fail | API key revoked/wrong | Check OPENAI_API_KEY in Railway variables |

### üü° WARNING (Degraded)

| Error Label | Symptom | Root Cause | First Action |
|-------------|---------|------------|--------------|
| `SQLITE_FALLBACK_ACTIVE` | Data wiped on deploy | PostgreSQL not connected | Check DATABASE_URL, PostgreSQL service status |
| `OPENAI_RATE_LIMIT` | 429 errors from OpenAI | Usage spike, quota exceeded | Enable retry with backoff, check usage |
| `PS101_STATE_CORRUPT` | Users seeing wrong questions | prompt_index out of range | Reset session PS101 state |
| `EMBEDDING_FAILED` | Semantic search not working | OpenAI embeddings API down | Use keyword fallback |

| Error Label | Symptom | Root Cause | First Action |

## Code Change Pre-Flight Checklist

**Before writing ANY code:**

```
ARCHITECTURE AWARENESS:
‚ñ° Do I understand what layer this touches? (DB / API / LLM / UI)
‚ñ° Have I checked for similar code patterns in the codebase?
‚ñ° Do I know what happens if this component fails?

DATABASE CHANGES:
‚ñ° Am I using context manager? (with get_conn() as conn:)
‚ñ° Am I using PostgreSQL syntax? (%s not ?, SERIAL not AUTOINCREMENT)
‚ñ° Am I getting cursor first? (cursor = conn.cursor())
‚ñ° Is this operation idempotent? (ON CONFLICT, check before insert)

ERROR HANDLING:
‚ñ° Am I logging errors explicitly? (not swallowing exceptions)
‚ñ° Will this fail gracefully? (fallback behavior defined)
‚ñ° Can I diagnose this from logs alone? (enough context logged)

DEPLOYMENT SAFETY:
‚ñ° Can I rollback this change? (git revert path clear)
‚ñ° Is there a feature flag? (can disable without deploy)
‚ñ° Have I tested locally? (golden dataset, manual test)
‚ñ° Did I check for breaking changes? (API contracts, schema)
```

## Symptom ‚Üí Diagnosis Flow

### Symptom: "Invalid Credentials" After Deployment

```
1. Check: Is PostgreSQL connected?
   railway logs | grep STORAGE
   ‚Üí Look for: "[STORAGE] ‚úÖ PostgreSQL connection pool created"
   ‚Üí If SQLite fallback: DATABASE_URL issue

2. Check: Does user exist in database?
   # This requires Railway shell or psql access
   ‚Üí If user missing: Database wiped (SQLite fallback confirmed)

3. Check: Was deployment recent?
   git log --oneline -5
   ‚Üí If deploy within last hour: Database reset on deploy

4. Diagnosis: SQLITE_FALLBACK_ACTIVE
   ‚Üí Action: Fix DATABASE_URL (ensure railway.internal)
   ‚Üí Verify: PostgreSQL service status in Railway
```

### Symptom: Deployment Failed

```
1. Check: Build logs or deploy logs?
   Railway dashboard ‚Üí Deployment ‚Üí Build Logs vs Deploy Logs
   ‚Üí If no deploy logs: App crashed on startup
   ‚Üí If build failed: Dependency or syntax error

2. Check: What's the error message?
   Look for:
   - "AttributeError" ‚Üí Likely context manager bug
   - "ModuleNotFoundError" ‚Üí Missing dependency in requirements.txt
   - "psycopg2" ‚Üí PostgreSQL connection issue
   - "Syntax Error" ‚Üí Python syntax error in code

3. Check: Recent commits
   git log --oneline -3
   ‚Üí Identify what changed recently

4. Action: Rollback if needed
   git revert HEAD
   git push railway-origin main
```

### Symptom: App Slow / Timeouts

```
1. Check: Health endpoint
   curl https://what-is-my-delta-site-production.up.railway.app/health
   ‚Üí Check p95_latency_ms

2. Check: Is it OpenAI/Anthropic?
   Railway logs | grep -i "openai\|anthropic"
   ‚Üí Look for slow response times

3. Check: Database queries
   ‚Üí Slow queries (missing indexes)
   ‚Üí Too many queries (N+1 problem)

4. Diagnosis:
   ‚Üí If AI calls: Add timeout, retry logic
   ‚Üí If DB: Add indexes, optimize queries
   ‚Üí If overall: Scale Railway instance
```

### Symptom: Users Seeing Old Data

```
1. Check: Browser cache?
   ‚Üí Ask user to hard refresh (Ctrl+Shift+R)

2. Check: Netlify deployment
   ‚Üí Is frontend deployed?
   ‚Üí Check Netlify dashboard

3. Check: Session state
   ‚Üí Old session data in localStorage
   ‚Üí Clear localStorage in browser console

4. Diagnosis: LIKELY_CACHING
   ‚Üí Action: Add cache-busting headers
   ‚Üí Or: Increment version number in /config
```

## Code Pattern Filters

### ‚úÖ CORRECT Patterns (Use These)

```python
# Database operations
with get_conn() as conn:
    cursor = conn.cursor()
    cursor.execute("SELECT * FROM users WHERE email = %s", (email,))
    result = cursor.fetchone()

# Error handling with logging
try:
    result = risky_operation()
except SpecificException as e:
    logger.error(f"Operation failed: {e}")
    # Explicit fallback or re-raise
    raise

# Idempotent inserts
cursor.execute("""
    INSERT INTO table (id, data)
    VALUES (%s, %s)
    ON CONFLICT (id) DO UPDATE SET data = EXCLUDED.data
""", (id, data))

# Retry with backoff
@retry_with_exponential_backoff(max_retries=3)
def call_external_api():
    # API call here
    pass
```

### ‚ùå INCORRECT Patterns (Never Use)

```python
# Wrong: Direct get_conn() usage
conn = get_conn()
cursor = conn.execute(...)  # AttributeError!

# Wrong: SQLite syntax in PostgreSQL
cursor.execute("SELECT * FROM users WHERE email = ?", (email,))  # Use %s

# Wrong: Silent exception swallowing
try:
    risky_operation()
except:
    pass  # No logging, no fallback

# Wrong: Non-idempotent operations
cursor.execute("INSERT INTO table VALUES (%s)", (data,))  # Fails on duplicate

# Wrong: Infinite retry
while True:
    try:
        call_api()
        break
    except:
        continue  # Never gives up!
```

## Debugging Workflow

### Step 1: Classify the Issue

```
Ask yourself:
1. When did this start? (recent deploy? always?)
2. Who is affected? (all users? one user? specific action?)
3. What's the error? (exception? wrong behavior? slow?)
4. Where is it? (frontend? API? database? LLM?)

Map to category:
- INFRA: Railway, PostgreSQL, networking
- DATA: Sessions, users, PS101 state
- MODEL: OpenAI, Anthropic, prompts
- PROMPT: CSV, JSON parsing
- INTEGRATION: Job sources, external APIs
```

### Step 2: Gather Context

```
Collect diagnostics:
‚ñ° Railway deployment logs (last 200 lines)
‚ñ° Health endpoint response (/health)
‚ñ° Recent git commits (git log -5)
‚ñ° Environment variables (railway variables)
‚ñ° PostgreSQL service status
‚ñ° Error message (full traceback)
‚ñ° User actions that triggered it
‚ñ° Recent deployments timeline
```

### Step 3: Form Hypothesis

```
Based on error category + context:

If INFRA error:
  ‚Üí Check DATABASE_URL format
  ‚Üí Check PostgreSQL service status
  ‚Üí Check recent Railway changes

If DATA error:
  ‚Üí Check database schema version
  ‚Üí Check for foreign key violations
  ‚Üí Check session expiration

If MODEL error:
  ‚Üí Check API keys set
  ‚Üí Check rate limits / quota
  ‚Üí Check model names valid

If PROMPT error:
  ‚Üí Validate JSON/CSV files
  ‚Üí Check file permissions
  ‚Üí Check for schema changes

If INTEGRATION error:
  ‚Üí Check external API status
  ‚Üí Check API keys / auth
  ‚Üí Check for API changes
```

### Step 4: Test Hypothesis

```
Verify hypothesis:
‚ñ° Can I reproduce locally?
‚ñ° Does the log match the hypothesis?
‚ñ° Does a simple fix resolve it?
‚ñ° Is there a known playbook for this?

If hypothesis wrong:
‚Üí Go back to Step 2 with more context
‚Üí Consult SELF_DIAGNOSTIC_FRAMEWORK.md
‚Üí Escalate to NARs with full context
```

### Step 5: Implement Fix

```
Before fixing:
‚ñ° Check pre-flight checklist (above)
‚ñ° Ensure rollback path exists
‚ñ° Test locally first
‚ñ° Run golden dataset tests

After fixing:
‚ñ° Deploy with clear commit message
‚ñ° Monitor logs for 5 minutes
‚ñ° Verify health endpoint
‚ñ° Test the specific symptom
‚ñ° Update documentation if new pattern
```

## Emergency Procedures

### üö® Production is Down

```
IMMEDIATE:
1. Check Railway dashboard - is service running?
2. If crashed: Check deploy logs for error
3. If context manager bug: Rollback immediately
4. If PostgreSQL: Check DATABASE_URL, PostgreSQL service

ROLLBACK:
git revert HEAD
git push railway-origin main --force
# Wait 2 minutes
curl /health

COMMUNICATE:
- Post status update
- ETA for fix
- Workaround if any
```

### üî• Data Loss Detected

```
ASSESS:
1. Is PostgreSQL connected? (check logs for STORAGE messages)
2. If SQLite fallback: All data lost on last deploy
3. If PostgreSQL: Check if table dropped/truncated

RECOVER:
- If SQLite: No recovery possible (ephemeral)
- If PostgreSQL: Check for backups in Railway
- Last resort: Restore from git history + redeploy

PREVENT:
- Ensure DATABASE_URL uses railway.internal
- Verify PostgreSQL service active
- Add data backup strategy
```

### ‚ö° Performance Degradation

```
DIAGNOSE:
1. Check /health endpoint - what's p95_latency_ms?
2. Check Railway metrics - CPU/Memory usage
3. Check logs - slow queries, API timeouts?

IMMEDIATE FIX:
- If OpenAI timeout: Add shorter timeout + fallback
- If DB slow: Add indexes, optimize queries
- If memory: Restart service, investigate leak

LONG TERM:
- Scale Railway instance
- Add caching layer
- Optimize hot paths
```
